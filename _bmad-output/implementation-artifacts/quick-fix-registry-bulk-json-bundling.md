# Quick Flow: Fix incomplete registry seed (sources-bulk.json bundling bug)

## Intent

Immediately after merging "wire registry into serve startup" and deploying to
production, live verification (`get_runtime_logs` on the real Vercel deployment)
showed `[seedRegistryIfEmpty] seeded 247 tier-0 sources` instead of the expected
~1,024. Production is right now under-seeded and already started crawling with
incomplete data.

## Root cause (confirmed)

`packages/radar-engine/src/registry/assemble.ts`'s `loadBulkJson()` reads
`sources-bulk.json` (429KB, 847 generated entries — state arts councils,
festivals, residencies) via `readFileSync` at a path resolved from
`dirname(fileURLToPath(import.meta.url))`. This is the EXACT same class of bug
fixed twice already today (Postgres schema `.sql` files) — Next.js/Turbopack's
serverless bundler traces static JS imports, not files resolved dynamically at
runtime, so `sources-bulk.json` doesn't reliably end up in the deployed
function bundle. `loadBulkJson()` silently returns `[]` when the file isn't
found (no error) — so production only got the 3 hand-curated TS bundles
(LITERARY_FICTION_SOURCES + POETRY_SOURCES + CNF_SOURCES, ~386 entries,
~247 after tier-0/dedup filtering), missing all 847 bulk-generated entries.

Confirmed via a codebase-wide sweep: this is the only remaining occurrence of
this bug pattern in the actual runtime path (`export-json.ts` also uses
`fileURLToPath(import.meta.url)` but it's a dev-only export script never
called by the running app — leave it alone).

## Approach (leader-decided, do not redesign)

### 1. Convert sources-bulk.json into a statically-importable TS module

Run this exact one-off conversion (from `packages/radar-engine/src/registry/`):

```bash
node -e "
const fs = require('fs');
const data = JSON.parse(fs.readFileSync('sources-bulk.json', 'utf8'));
const out = \"import type { SourceRegistryEntry } from './types.js';\n\n// Auto-generated by scripts/generate-registry.mjs -- do not hand-edit.\nexport const BULK_SOURCES: SourceRegistryEntry[] = \" + JSON.stringify(data.sources, null, 2) + ';\n';
fs.writeFileSync('sources-bulk.ts', out);
"
```

Then delete `sources-bulk.json` (single source of truth — no drift risk between
a hand-maintained-in-sync `.json` and `.ts`).

### 2. Update `assemble.ts`

Replace `resolveBulkJsonPath()` + `loadBulkJson()`'s file-reading logic with a
static import: `import { BULK_SOURCES } from './sources-bulk.js';` and use
`BULK_SOURCES` directly wherever `loadBulkJson()` was called. Remove the now-
unused `readFileSync`, `existsSync`, `fileURLToPath`, `dirname`, `join`,
`__dirname` — check the whole file first; only remove what's genuinely unused
after this change (this file may still need `join`/`dirname` for nothing else,
verify before deleting imports).

### 3. Update `scripts/generate-registry.mjs`

Change its output step (currently `writeFileSync(OUT, JSON.stringify(registry, null, 2))`
writing to `sources-bulk.json`) to instead write `sources-bulk.ts` in the same
format as step 1's conversion, so future regeneration (if the registry is ever
rebuilt) produces the correct static-import-compatible file, not the old
runtime-read JSON file. Read the full script first to understand `registry`'s
shape (likely `{ sources: [...], ... }` — only the `.sources` array matters for
the `.ts` output, matching what step 1's conversion did).

### 4. Make `seedRegistryIfEmpty` self-healing against the current bad production state (IMPORTANT — design change from the original version)

The version shipped this session gates on `engine.store.sources.size > 0` —
this means simply deploying the bugfix will NOT self-correct the already-live
production database, which currently has 247 sources (`size > 0` is already
true, so the idempotency guard would skip re-seeding forever, permanently
stuck at 247/1024 unless someone manually clears the `radar_sources` table).

Redesign `seedRegistryIfEmpty` to be dedup-aware instead of a blunt
size-check, so it naturally self-heals *any* partial-seed state (today's
incident, or any future one) without ever creating duplicates:

```ts
import { assembleRegistry, filterSources, type RadarEngine } from '@missa/radar-engine';

function normalizeUrl(url: string): string {
  return url.replace(/\/$/, '').toLowerCase();
}

export function seedRegistryIfEmpty(engine: RadarEngine): { loaded: number } | null {
  const existingUrls = new Set(
    [...engine.store.sources.values()].map((s) => normalizeUrl(s.url)),
  );
  const registry = assembleRegistry();
  const entries = filterSources(registry, { maxTier: 0 });

  let loaded = 0;
  for (const entry of entries) {
    const key = normalizeUrl(entry.url);
    if (existingUrls.has(key)) continue;
    engine.addSource({
      name: entry.name,
      url: entry.url,
      kind: entry.kind,
      checkIntervalHours: entry.checkIntervalHours,
    });
    existingUrls.add(key);
    loaded++;
  }

  if (loaded === 0) return null;
  console.log(`[seedRegistryIfEmpty] seeded ${loaded} new tier-0 sources from the opportunity-source registry`);
  return { loaded };
}
```

Check whether `assembleRegistry`/`filterSources` are already exported from
`@missa/radar-engine`'s top-level `src/index.ts` (they should be — verify, and
add the export if for some reason it's missing).

Rename/adjust the function's doc comment (it's no longer strictly "if empty" —
it's "seed anything from the registry not already present by URL"). Keep the
exported name `seedRegistryIfEmpty` for now (minimize unrelated churn to
`index.ts`'s export list and any other reference) but update its JSDoc to
describe the real (dedup-based) behavior accurately.

### Resolution: leader decided Option A (2026-07-07)

Reasoning (leader's, recorded verbatim for the record): `OpportunityType` is used generically everywhere it appears (tracker field, extractor keyword array, registry helpers) — there is no closed UI label mapping or exhaustive `switch` statement anywhere in the codebase that assumes a fixed set of values, so extending the union is low-risk. `'exhibition'` and `'commission'` are real, distinct, commonly-used terms in the visual-arts opportunity space (distinct from generic `'open-call'`/`'grant'`), and visual arts is one of the larger discipline groups in this registry — preserving that real-world distinction (rather than silently downgrading these 5 sources to generic types, which is what Option B would have done) is worth two extra union members.
