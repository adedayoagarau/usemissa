# Missa — Product Strategy & Idea Dump

Below is a strong feature map for your product as an **end-to-end submission platform** — not just applications, not just magazines, not just grants.
Think of the product as:
**Discovery → Submission → Intake → Review → Decision → Communication → Delivery → Archive/Analytics**
Competitors already cover pieces of this: Submittable supports publishing/journal submission workflows, review collaboration, and post-submission editing; SurveyMonkey Apply emphasizes workflow stages, labels, auto-bucketing, audit trails, review portals, and automations; OpenWater positions around awards, grants, scholarships, abstracts, reviews, automation, and reporting. Your opportunity is to connect the full lifecycle more elegantly. ([Submittable](https://www.submittable.com/solutions/publishing-and-journals?utm_source=chatgpt.com))

# **One-Line Product Definition**
**Missa is an end-to-end platform for discovering, collecting, reviewing, deciding, communicating, and delivering submissions of any kind.**
# **Strong Homepage Positioning**
**Manage every submission from open call to final outcome.**
Subheadline:
**Missa helps magazines, grantmakers, awards programs, festivals, fellowships, and organizations collect submissions, manage review, send decisions, and deliver final outcomes in one workflow.**
  
Yes — for an **enterprise-grade submission platform**, the product should not be designed as “one organization, one account.” It should be designed as a **multi-entity enterprise system** where each customer can manage multiple brands, departments, programs, legal entities, and reviewer groups under one enterprise account.
Also, on the screenshot: you do **not** need the legal Delaware company name to exactly match the product name. You can form as something like **Missa Flow, Inc.**, **Missa Platform, Inc.**, or a more neutral parent-company name, while marketing the product as **Missa**. Trademark and domain clearance should be checked separately.

# **Recommended Seat Model**
You want seats to be assigned at the **enterprise level**, but usable at the **entity level**.
## **Seat Types**
|  |  |  |
| :- | :- | :- |
| **Seat Type** | **Purpose** | **Example Users** |
| **Enterprise Admin** | Controls the whole account | CIO, operations lead, platform owner |
| **Entity Admin** | Manages one entity | Magazine editor, grant program director |
| **Program Manager** | Runs specific calls or programs | Awards manager, fellowship coordinator |
| **Reviewer** | Scores/reviews submissions | Judges, readers, committee members |
| **Finance Seat** | Handles fees, payouts, refunds, award amounts | Finance team |
| **Legal/Compliance Seat** | Reviews agreements, consent, records | Legal, compliance, grants admin |
| **Viewer/Read-only Seat** | Can view dashboards and exports | Executives, board members |
| **External Collaborator** | Limited guest access | Guest judge, partner reviewer |

For enterprise pricing, I would separate users into:
|  |  |
| :- | :- |
| **Category** | **Billing Treatment** |
| **Core seats** | Paid full seats: admins, managers, finance, legal |
| **Reviewer seats** | Lower-cost or usage-based seats |
| **External reviewers** | Free or pooled up to a limit, depending on plan |
| **Submitters** | Not paid seats |
| **Viewers** | Free or low-cost seats |

This matters because magazines, awards, and grant programs can have a lot of reviewers. Charging every reviewer as a full SaaS seat may create friction.

# **Permission Architecture**
Do **not** use a simple role system only. Enterprise customers need **RBAC + scoped permissions**.
## **Role-Based Access Control**
Roles define what someone can do.
Examples:
  - Create opportunities
  - Edit forms
  - View submissions
  - Assign reviewers
  - Review submissions
  - See submitter identity
  - Send emails
  - Make decisions
  - Export data
  - View financials
  - Manage users
  - Configure integrations
## **Scope-Based Access Control**
Scopes define where they can do it.
A user may have access to:
  - Entire enterprise
  - One entity
  - One program
  - One opportunity
  - One review round
  - One submission
  - One report
Example:
Sarah is an **Entity Admin** for “Emerging Writers Magazine,” but only a **Reviewer** for “Visual Arts Awards.”
This is essential for enterprise customers because one person may have different responsibilities across different entities.

# **1. Core Concept: Open Calls Are Containers**
An **open call** is the public-facing opportunity.
Examples:
  - “Spring 2027 Poetry Issue”
  - “Emerging Artist Grant”
  - “Annual Fiction Prize”
  - “Film Festival Submissions”
  - “Call for Conference Abstracts”
  - “Residency Applications”
  - “Startup Accelerator Cohort”
  - “Public Art RFP”
  - “Award Nominations”
But each open call may contain many internal paths.
Example:
Open Call: Spring 2027 Literary Issue
├── Type: Magazine Submission
├── Genres:
│   ├── Poetry
│   ├── Fiction
│   ├── Nonfiction
│   ├── Translation
│   └── Hybrid
├── Rules:
│   ├── Poetry: up to 5 poems
│   ├── Fiction: 1 story, max 7,500 words
│   ├── Nonfiction: 1 essay, max 5,000 words
│   └── Translation: original + translation required
└── Review Workflows:
    ├── Poetry goes to poetry readers
    ├── Fiction goes to fiction readers
    └── Translation goes to translation editor
So the system should not treat an open call as a single flat form. It should treat it as a **configurable intake object with nested submission paths**.

# **2. Distinguish Type, Genre, Category, Track, and Medium**
These terms should not be mixed together.
## **Recommended Definitions**
|  |  |  |
| :- | :- | :- |
| **Concept** | **Meaning** | **Example** |
| **Type** | What kind of opportunity this is | Magazine submission, grant, award, fellowship, contest, abstract, pitch |
| **Genre** | Creative or content classification | Poetry, fiction, nonfiction, photography, film, sculpture |
| **Category** | Admin-defined grouping inside a call | Emerging writer, student, professional, international |
| **Track** | A workflow or review lane | General track, invited track, youth track, regional track |
| **Medium** | Format of the submitted work | Text, image, video, audio, PDF, dataset, code, portfolio |
| **Discipline** | Broader field | Literature, visual arts, film, music, science, education |
| **Deliverable Type** | What the accepted submitter must deliver | Final manuscript, grant report, tax form, contract, headshot |

This distinction matters because a single opportunity can have multiple dimensions.

# **5. Submission Package vs. Submission Item**
This is very important.
A submission is often not just one file.
You need two levels:
## **A. Submission Package**
The full thing the submitter sends.
Example:
“Ade’s submission to Spring 2027 Issue”
Contains:
  - Submitter profile
  - Contact info
  - Eligibility answers
  - Payment
  - Cover letter
  - Consent
  - One or more submitted works
  - Messages
  - Review status
  - Decision
## **B. Submission Item**
Each individual work inside the submission.
Example:
Submission Package: Ade’s Poetry Submission
├── Item 1: Poem A
├── Item 2: Poem B
├── Item 3: Poem C
├── Item 4: Poem D
└── Item 5: Poem E
This solves a huge edge case.
For magazines, a person may submit **five poems** in one submission, but the editors may accept one, reject three, and ask for revisions on one.
So decisions may happen at two levels:
|  |  |
| :- | :- |
| **Level** | **Example** |
| **Package-level decision** | Submission accepted, declined, waitlisted |
| **Item-level decision** | Poem 2 accepted, Poem 4 held, Poem 5 declined |

This is critical for magazines, portfolios, art submissions, film bundles, abstract collections, and multi-work contest entries.

# **7. Edge Cases You Should Design For**
## **1. One Open Call, Many Genres**
Example:
A magazine accepts poetry, fiction, nonfiction, art, and translation in one call.
Solution:
  - Use **Submission Paths**
  - Each path has its own form, rules, files, fees, reviewers, and workflow.
-----
## **2. One Submission, Multiple Works**
Example:
A poet submits five poems.
Solution:
  - Use **Submission Package** and **Submission Items**
  - Allow item-level metadata, review, and decisions.
-----
## **3. Multiple Genres in One Submission**
Example:
A hybrid artist submits text, images, and audio.
Solution:
  - Allow multiple taxonomy tags per item.
  - Allow admins to permit or block mixed-genre submissions.
Submission Item:
\- Title: “Field Notes”
\- Genres: Hybrid, Essay, Visual Art
\- Media: PDF, Images, Audio
-----
## **4. Simultaneous Submissions**
Example:
A writer submits the same story elsewhere and needs to withdraw it if accepted.
Solution:
  - Add submitter-facing setting: “This is a simultaneous submission.”
  - Allow full or partial withdrawal.
  - Track withdrawal reason.
  - Notify admins automatically.
-----
## **5. Partial Withdrawal**
Example:
A poetry packet contains five poems, but one poem was accepted elsewhere.
Solution:
  - Let submitter withdraw individual items, not just the full submission.
Statuses:
Poem 1: Under review
Poem 2: Withdrawn
Poem 3: Under review
Poem 4: Accepted
Poem 5: Declined
-----
## **6. Anonymous or Blind Review**
Example:
A contest requires blind judging.
Solution:
  - Hide identifying fields from reviewers.
  - Separate submitter metadata from review packet.
  - Detect identifying info in uploaded files where possible.
  - Allow admin-only identity access.
Blind review should be configurable by path, not just by whole open call.
-----

Support:

# **7. Expected Response Window**
This is a killer feature.
Missa should tell users:
**“You’ll probably hear back around…”**
## **How to Calculate It**
Use multiple signals:
|  |  |
| :- | :- |
| **Signal** | **Example** |
| Organization historical response time | Median response: 74 days |
| User-reported tracker data | 300 tracked outcomes |
| Missa-hosted workflow stage | Review closes on March 1 |
| Past cycle behavior | Last year decisions went out 2 weeks after deadline |
| Submission type | Grants usually slower than magazine declines |
| Stated timeline | “Winners announced in June” |
| Current stage | “In final review” |

## **Output Format**
Do not pretend to know the exact date. Use ranges.
Expected response: late March to mid-April
or
Most users hear back in 60–90 days.
You submitted 47 days ago.
or
This organization says decisions will be announced in June.
## **Confidence Levels**
High confidence
Based on 430 tracked responses and current Missa workflow data.
Medium confidence
Based on historical user-reported data.
Low confidence
Based on the organization’s stated timeline only.
## **Happy Content**
You’re still within the normal response window.
Most responses for this opportunity arrive between 60 and 90 days.
## **Unhappy Content**
This one is taking longer than usual.
You submitted 126 days ago. Similar submissions usually hear back within 90 days.
Then offer:
\[Mark as no response\] \[Set follow-up reminder\] \[Archive\] \[Add note\]

# **The actual product**
## **1. Opportunity Inbox**
This is the home feed.
Missa proactively surfaces opportunities:
New for you
Opening soon
Closing soon
Recently updated
Saved by you
From organizations you follow
Detected from your email
Recommended based on your profile
User actions:
\[Save\]
\[Track\]
\[Add to calendar\]
\[Start preparing\]
\[Hide\]
\[Not relevant\]
The inbox should feel like:
“Here are the opportunities you should care about, already organized.”
-----
## **2. Auto Calendar**
Every saved or tracked opportunity should create calendar intelligence.
Features:
  - Add deadline to Missa calendar
  - Sync to Google Calendar, Apple Calendar, Outlook
  - Add opening date
  - Add closing date
  - Add internal prep reminders
  - Add expected response window
  - Add delivery deadlines after acceptance
  - Add recommender/reference deadlines
  - Add follow-up reminders
Example:
North River Review
Deadline: March 15
Prep reminder: March 8
Expected response: late May–June
This is one of the most emotionally useful features.
-----
## **3. Email Reminder Engine**
Not just generic reminders. Contextual reminders.
Examples:
This opportunity closes in 7 days.
You saved it but have not started preparing.
You selected 3 poems, but this magazine allows up to 5.
You submitted 82 days ago. Similar responses usually arrive around now.
This fellowship opens next week based on last year’s cycle.
You were accepted. Final materials are due Friday.
The reminder engine should be the nervous system of the product.
-----
## **4. Gmail Auto-Tracking**
This is the “tracker that updates itself” feature.
When the user connects Gmail, Missa can detect:
|  |  |
| :- | :- |
| **Email type** | **Tracker action** |
| “Thank you for submitting” | Mark submitted |
| “Your application was received” | Mark received |
| “Your submission is under review” | Mark in review |
| “Congratulations” | Suggest accepted/finalist |
| “We regret to inform you” | Suggest declined |
| “Deadline extended” | Update deadline |
| “Applications now open” | Add to Opportunity Inbox |
| “Please sign agreement” | Create delivery task |
| “Final materials due” | Create checklist/reminder |

Important: sensitive changes should be confirmed first.
Example:
Missa found a possible decision email.
North River Review
Suggested status: Declined
\[Confirm\] \[Edit\] \[Ignore\]
This preserves trust.

# **7. Expected Response Windows**
This is a killer feature.
Instead of just tracking status, Missa tells users:
You may hear back around late March to mid-April.
Based on:
  - Historical response data
  - Missa-hosted review workflows
  - User-tracked outcomes
  - Organization-stated timelines
  - Past cycles
  - Similar opportunities
States:
Too early
Still normal
Likely soon
Taking longer than usual
Very overdue
No response candidate
Example:
You’re still within the normal response window.
Most tracked responses from this organization arrive between 60 and 90 days. You submitted 43 days ago.
This reduces anxiety. That matters.

# **8. Lifecycle management**
The product should not end at “submitted.”
It should manage the full lifecycle:
Discover
→ Save
→ Prepare
→ Submit
→ Track
→ Wait
→ Respond
→ Accept / Decline / Withdraw
→ Deliver final materials
→ Archive
→ Reuse materials
→ Submit again
## **After acceptance**
Missa should create tasks like:
  - Sign agreement
  - Upload final file
  - Submit bio
  - Submit headshot
  - Provide payment details
  - Confirm rights
  - Submit tax form
  - Complete final report
  - Add publication date
  - Mark delivered
## **After rejection**
Missa should help recover:
This submission was declined.
We updated your tracker. The work is now available to send elsewhere.
Actions:
\[Find similar opportunities\]
\[Submit this work again\]
\[Archive\]
\[Add private note\]

# **8. The Edge Cases Missa Should Own**
This is where competitors often get clumsy.
## **Creative / Magazine Edge Cases**
  - One call accepts many genres.
  - One submission contains multiple poems.
  - One poem is withdrawn but the rest stay active.
  - One item is accepted, another is declined.
  - A submitter revises a piece.
  - A work is simultaneous elsewhere.
  - A call is blind but admins need identity access.
  - A category closes early after reaching cap.
  - A submitter pays for one category but has a waiver for another.
  - A magazine needs accepted work to move into publication production.
## **Grant / Enterprise Edge Cases**
  - Organization submits with multiple collaborators.
  - Fiscal sponsor required.
  - References/recommendations are confidential.
  - Eligibility screening blocks applicants.
  - Review happens in stages.
  - Board approval happens after panel review.
  - Award amount differs from requested amount.
  - Grantee must submit interim/final reports.
  - Finance needs payment milestones.
  - Legal needs signed agreements.
## **Awards / Festival Edge Cases**
  - Entry can be nominated by someone else.
  - Nominee must accept nomination.
  - Judges have conflicts.
  - Public voting runs alongside jury review.
  - Finalists get different email than winners.
  - Winner page must publish later.
  - Official selection needs laurels, press kit, screening copy, or certificate.
## **Conference / Abstract Edge Cases**
  - Multiple authors.
  - Presenting author differs from submitting author.
  - Abstract can become poster, paper, talk, panel, or workshop.
  - Accepted abstract needs scheduling.
  - Speaker needs registration/ticketing.
  - Proceedings/abstract book must be generated.

# Technical stack...

# **Honest technical stack for Missa**
My honest recommendation:
**Build Missa as a TypeScript-first modular monolith with durable background workflows, Postgres as the source of truth, S3-compatible file storage, strong enterprise auth, and a separate Radar/import worker system.**
Do **not** start with microservices.  
Do **not** start with a huge event-streaming architecture.  
Do **not** build your own auth, email infrastructure, payments infrastructure, or workflow engine.
Missa is complicated because of **workflow depth**, not because it needs hyperscale architecture on day one.

# **5. Database**
## **Pick Postgres**
Use:
# **PostgreSQL as the source of truth**
Postgres should own:
  - Enterprises
  - Entities
  - Programs
  - Open calls
  - Submission paths
  - Forms
  - Submissions
  - Submission items
  - Reviews
  - Decisions
  - Messages
  - Files metadata
  - Seats
  - Roles
  - Permissions
  - Audit logs
  - Payments metadata
  - Radar opportunities
Postgres is especially appropriate because Missa is relational and workflow-heavy. You will constantly need queries like:
“Show all submissions in this entity, for this call, in these genres, assigned to these reviewers, with these statuses, excluding conflicts, grouped by decision stage.”
That is a relational problem.
## **Multi-tenancy design**
Use:
Single database
Single schema
tenant/account/entity IDs on every relevant row
app-level authorization
Postgres Row-Level Security where appropriate
PostgreSQL supports row-level security policies that can restrict which rows are returned, inserted, updated, or deleted by normal queries. That makes it useful as a defense-in-depth layer for multi-tenant systems, though you should still enforce authorization in the application layer too. ([PostgreSQL](https://www.postgresql.org/docs/current/ddl-rowsecurity.html?utm_source=chatgpt.com))
## **ORM**
My honest pick:
# **Drizzle or Kysely**
Use Drizzle if you want:
  - Type-safe schema
  - SQL-like control
  - Good migration discipline
  - Less ORM magic
Use Kysely if your team is very SQL-comfortable.
I would avoid going too deep into ORM abstraction because Missa will need serious reporting, filtering, permission checks, and analytics queries.
