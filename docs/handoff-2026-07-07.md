## ⚠️ Vocabulary leak found in the new backend's UI — not yet fixed

`packages/radar-engine/src/server/ui.ts` is a minimal server-rendered single-page UI. Its `<title>` and visible `<h1>` both literally read **"Missa Radar"** (line 10 and line 44) — this is exactly the internal engine name the copy audit banned from user-facing surfaces on the landing page (mapped to "Alerts" there). This file wasn't in scope for that audit because it didn't exist yet when the audit ran. Recommend renaming to something like "Missa" or "Missa Tracker" for the title/H1, consistent with the landing page's `Radar → Alerts` rule — but confirm with the user first since this app may have its own intended internal/admin audience (unclear from a first pass whether this UI is meant for end users or as a dev/ops dashboard).

Everything else checked in that file/package came back clean: `FitLevel` is already word-based (`strong | possible | weak | not-eligible | unknown`), matching the approved Match-tier mapping, not a raw score. Field names like `trustSignals`/`radarProfiles` in `server.ts` are internal data-layer identifiers, not displayed labels — fine to keep per the "internal names can stay" rule. This was a quick grep-based check, not a full read of the package — worth a proper pass before shipping.
